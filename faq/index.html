Frequently Asked Questions

1. What is performance testing?

Performance testing is the process of discovering the operational characteristics of a system or application under a workload. Capability, capacity, stability, and speed are all considered. The core metrics are response times, throughput and resource-utilization, though other metrics, such as mean-time between failure, and operating and infrastructure costs, can also be derived through performance testing. It explores several system qualities, simplified to:

  • Actual Speed: How quickly does the system respond under load?
  • Perceived Speed: Does the system meet the end user’s performance expectations?
  • Capacity: Is the infrastructure adequately sized?
  • Stability: Does the system behave correctly under load?
  • Scalability: Can the system grow to handle future volumes?

2. What are the types of performance tests?

There are four families of performance tests:

  • Ad hoc tests: A quick, simple test run by developers to identify obvious performance issues such as infinite loops and memory leaks.
  • Stress tests: Pushes the application and/or infrastructure beyond specifications to see when and how it might fail. Stress tests involve operating systems beyond their intended capacity. Common types of stress tests are:
    • Capacity test: Incrementally increases the number of concurrent users while monitoring the system resource load. The goal is to identify the number of concurrent connections at which the system performs at ideal capacity. Ideal capacity is a business decision that generally ranges from 60% to 80% of total capacity.
    • Saturation test: A continuation of the capacity test. Discovers the point at which a resource has reached full utilization.
    • Spike test: Determines the performance characteristics of the system under test when subjected to rapid increases in workload for short periods of time.
  • Load tests: Places a certain load on a system to discover how the system performs over a long term. It’s ideal to use about 80% of the maximum load the system can handle. Common types of load tests are:
    • Benchmark test: Compares performance at specific points in the project life cycle. A repeatable test, the most common use is comparing performance between nightly or weekly builds and between major releases before they are pushed to production.
    • Reliability: Determines how the system handles a realistic load over an extended period of time, sometimes up to 30 days. The length of time is critical in catching the effects of background processes on operational performance. These processes include log rotation, database backup and replication. Reliability tests are also referred to as endurance tests.
    • Availability: Determines how quickly a system recovers from failure. When a critical performance issue is identified by any of the other tests, availability testing can determine how much time it takes a system to return to normal operating conditions.
    • Scalability: Tests the ability of a system or application to handle additional workload while maintaining the expected performance levels. If a particular resource is increased (ten front-ends instead of five, or 16GB of RAM rather than 8GB), how does this affect performance?
  • UI/Rendering tests: Judges the speed and responsiveness of the client side user interface. This is the perceived speed of the application from the end user’s perspective. These can also be considered functional tests, however the perceived performance of an application is critical to the success of any customer-facing project and must be considered as part of the overall performance.

3. Why should I test performance?

Performance testing is about assessing risk. Your company probably wants to identify performance and operational issues before your customers do. Quantifying and forecasting these issues is determined through proper performance testing. Some common issues include:

  • Configuration issues: What is the ideal load-balancer configuration? How much memory should be dedicated to caching? How often can the database be replicated without affecting response times?
  • Software issues: These include not only the code developed in house but also any open-source or vendor software dependencies.
  • Legal issues: Determine and meet your service level agreements for partners and customers as required. Verify, as needed, that your vendors will meet their SLAs while your system is under load.
  • Business criteria: Determine your business response to client usage. For example, if the system must support 500 simultaneous transactions with average response time under two seconds and no higher than five, will it succeed?
  • Forecasting infrastructure cost: Are you purchasing too much hardware? Too little? When will you need to increase disk space?

4. When should performance testing be included in the project life cycle?

Performance testing is useful at all stages of the project life cycle. It's about risk assessment. The sooner issues are discovered, the sooner they can be resolved. Issues can be found not just in the software, but in the overall infrastructure as well. The earlier in the process performance testing is incorporated, the greater the chance the project will finish on time and on budget. There are several reasons:

  • Nightly or weekly tests: Discover and correct problems during development
  • Release candidate tests: Can provide data for production capacity planning
  • Pre-launch tests: Can identify hardware and network configuration issues before your customers do

5. What tools are available for testing the performance of web applications?

Traditionally, there are three types of tools available (outlined below).

Vendor tools

  • Advantages:
    • Generates realistic load scenarios
    • Results can be shared and verified
    • Training is available
  • Disadvantages:
    • Expensive acquisition, maintenance, and training costs
    • Acquisition can take weeks or months due to traditional sales cycles
    • Proprietary scripting languages require training
  • Examples: Gomez, LoadRunner, SilkPerformer, Rational Performance Tester

Open Source

  • Advantages:
    • Supports most vendor features
    • Results and can be shared outside company
    • Scripted in standard programming languages
  • Disadvantages:
    • Limited reporting functionality
    • Lack of support or training
    • Poor or no graphical interface
  • Examples: JMeter, Grinder, OpenSTA, httperf, Siege, Apache Bench

In House

  • Advantages:
    • Experts exist in house
    • Tests can be performed quickly
  • Disadvantages:
    • In-house knowledge may not exist and can be lost when the expert(s) leave the company
    • Validity of results questionable outside company (customers, partners, publications)
    • Requires dedicated resources for the lifetime of the tool
  • Examples: Usually one or more scripts written by the application developers

6. Which of the open source tools is the best?

We like httperf the best. It takes the most sensible approach to simulating complex workload scenarios. It doesn’t support having more than one load generating machine so for very large tests you’re better off with JMeter.

By the way, there are many more open source tools than what we list above (we stuck with the best ones).
